# 认知循环

Agent 的行为模型是一个**可在任意阶段回溯的循环**，不是瀑布流：

```
感知 ──→ 建模 ──→ 行动 ──→ 验证 ──→ 感知 ...
```

## 四个阶段

| 阶段 | 做什么 | 输入 | 输出 |
|------|--------|------|------|
| **感知** | 读取 soul + manifest，获取项目全貌和约束 | 文件系统 | 项目上下文模型 |
| **建模** | 拆解问题，形成方案 | 上下文 + 需求 | 执行计划 |
| **行动** | 执行方案（写代码、改配置） | 执行计划 | 产物 |
| **验证** | 跑验证命令，确认结果 | 产物 | 通过 / 回退 |

## 回溯机制

关键设计：验证失败时不是"修一下继续"，而是**回到感知阶段重新来过**。

这个区别很重要。"修一下继续"意味着你还在原来的心智模型里打补丁——如果模型本身就错了，补丁越打越偏。"回到感知"意味着你承认当前方案可能是错的，重新审视全貌后再决定怎么办。

```
感知 ──→ 建模 ──→ 行动 ──→ 验证
  ▲                         │
  │                    失败  │
  └─────────────────────────┘
```

## 在 Archon 中的体现

认知循环不是一个抽象概念——它落地到具体的文件和工作流中：

| 阶段 | 对应的 Archon 机制 |
|------|-------------------|
| 感知 | 每次唤醒加载 soul.md + manifest.md |
| 建模 | Plan 模式 / Demand 的决策关卡 |
| 行动 | Demand 的执行阶段 |
| 验证 | 验证门（lint + typecheck + test） |
| 回溯 | 验证失败 → 重新感知上下文 |

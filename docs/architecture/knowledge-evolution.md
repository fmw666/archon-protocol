# 知识进化系统

Archon 不是静态框架——它在每次交付中**积累和强化**自己的知识。

## 触发 → 捕获 → 沉淀

```
 触发信号                  沉淀路径                 成熟度升级
 ──────────               ──────────               ──────────
 撞墙换路 ─┐              L0 类型系统    ◀─ MUST    发现
 重复模式  │              L1 Lint 规则              │
 外部反馈  ├─→ 评估 ──→  L2 编辑器规则              ↓
 新技术引入 │              L3 技能文档    ◀─ SHOULD  SHOULD
 业务洞察  │              L4 架构决策               │
 约定成型 ─┘              L5 Manifest               ↓
                                                   MUST
```

## 触发条件

| 信号 | 典型沉淀 |
|------|---------|
| **撞墙换路**：方案 A 失败，切到 B | 推理胶囊（嵌入 skill）：A 的失败信号 → B 的推导路径 → B 的验证方式 |
| **重复模式**：第二次写相同结构的代码 | 编辑器规则，或抽象为组件/工具函数 |
| **外部反馈**：用户或 reviewer 指出问题 | 更新规则/技能，防止复发 |
| **新技术引入**：首次使用某库/模式 | 技能文档：推荐模式 + 禁止模式 + 已知陷阱 |
| **业务洞察**：从需求中发现可复用策略 | 架构决策或技能 |
| **约定成型**：隐含约定在代码中反复出现 | 编辑器规则：从"默契"变为"机器执行" |

**不触发**的场景：一次性 bug fix、纯文案修改、已有规则已覆盖的模式。避免沉淀噪音。

## 推理胶囊

撞墙换路时，把成功的调试/修复路径嵌入相关 skill 文档：

```
症状 → 根因 → 修法
```

不是独立文件，不是独立格式——**嵌入已有 skill**。

下次遇到同类症状，直接复用验证过的路径，不重新推理。这比"记住结论"更有价值——它记住的是**推理路径**。

## 反合理化表（Red Flags）

嵌入各 skill 文档中的对照表：

| 偷懒念头 | 现实 |
|---------|------|
| "太简单了不用测试" | 简单代码也会被依赖，回归成本不比复杂代码低 |
| "先交付后面补" | 后面不存在。Backlog 只会更长 |
| "这只是内部工具" | 内部工具的用户也是用户 |
| "这是临时方案" | 临时 = 永久减去文档 |

**设计原理**：Soul 的"品质洁癖"解决"知不知道该做"，Red Flags 解决"知道但找借口不做"。

## Stakeholder 对话记忆

会话制框架的天然缺陷——**每次启动都是白纸**。项目负责人应该记得与 stakeholder 讨论过什么、决定了什么。

### 记忆架构

```
 会话 N                                          会话 N+1
┌──────────────────────┐                        ┌──────────────────────┐
│ 用户: "要不要做 SSR？"  │                        │ 用户: "要不要做 SSR？"  │
│         ↓             │                        │         ↓             │
│   决策关卡 → 否决       │                        │     前置扫描           │
│         ↓             │                        │   命中备忘: SSR 否决    │
│   收尾 → 写入备忘       │                        │         ↓             │
│                       │    manifest.md         │  "上次讨论过，否决      │
│   SSR → 否决 ─────────┼──→ 对话备忘表 ────────→│   因为 X，无新变量"     │
└──────────────────────┘                        └──────────────────────┘
```

### 记忆分类

不同类型的记忆有不同的持久化载体：

| 类型 | 载体 | 生命周期 |
|------|------|---------|
| 交付记录 | drift.md | 按周期压缩 |
| 架构正向决策 | decisions.md (ADR) | 永久 |
| 架构否决决策 | decisions.md (Negative ADR) | 永久（含重审条件） |
| Stakeholder 结论 | manifest 对话备忘表 | 滚动 10 条 |

备忘表上限 **10 条**，超限时最旧条目：
- 仍有长期价值？→ **升格**到更强载体（ADR / skill）
- 已过时或被覆盖？→ 直接移除

## 递归适用

Archon 不只治理代码，也治理自己。Soul 中每条原则同时约束 Archon 框架自身：

- "品质洁癖" → 治理文件也不允许死代码、不一致的命名
- "新代码=新护栏" → 新增工程阶段同步补齐治理机制
- "零容忍死代码" → **过时的治理文件比过时的代码更危险**——代码会报错，治理文件只会误导

## 已知盲区模式

| 盲区 | 表现 | 对策 |
|------|------|------|
| **被动姿态** | 只在信号出现时才反应，不主动审视 | plan/review 强制三问 |
| **隧道视野** | 专注当前问题忽视周边 | "完整工作流我覆盖了哪一段？" |
| **文档优先于机制** | 默认写原则而非构建执行机制 | "谁来确保它被执行？" |
| **由内而外设计** | 从系统能力出发而非用户工作流 | 先画用户流程再看系统介入点 |

# 系统总览

Archon 是一组 Markdown 文件构成的协议——没有运行时，没有依赖。它通过**文件结构和命名约定**驱动 AI agent 的行为模式。

## 分层设计

Archon 按**可搬运性**分为两层：

### 协议层（项目无关）

可直接复制到任何新项目，不需修改：

| 文件 | 职责 |
|------|------|
| `soul.md` | 认知内核——身份基石、认知循环、进化机制 |
| `archon.md` | 统一入口 + 意图路由 |
| `archon-plan.md` | 规划模式工作流 |
| `archon-demand.md` | 需求交付工作流 |
| `archon-review.md` | 全量审查工作流 |
| `archon-reviewer.md` | 审查 sub-agent 协议 |
| `archon-capture-auditor.md` | 知识捕获 sub-agent 协议 |
| `archon-decouple` | 解耦守则 |
| `archon-wake` | 唤醒触发器 |

### 状态层（项目绑定）

每个项目独立，由 Archon 自动维护：

| 文件 | 职责 | 由谁更新 |
|------|------|---------|
| `manifest.md` | 项目全貌 | 主 Agent 每次交付后 |
| `drift.md` | 漂移计数 + 交付日志 | 主 Agent 每次交付后 |
| `debt.md` | 技术债务登记 | 主 Agent / Reviewer |

**唯一允许放项目细节的文件是 `manifest.md`。** 协议层文件中禁止出现具体技术名、文件路径、shell 命令、包名。

## 唤醒机制

Archon 的入口是**零认知负担**的——用户不需要记任何命令名：

```
用户                                  Archon
────                                  ──────
"hi archon, 加个通知系统"              archon-wake (always-applied rule)
     │                                检测唤醒词 → 加载 soul + manifest
     └──────────────────────────────→ archon.md 路由逻辑
                                      意图分析 → "具体需求" → demand
                                           │
"/archon demand: 加个通知系统"               │ (显式前缀，跳过意图分析)
     └──────────────────────────────→ archon.md → demand
                                              │
"/archon-demand 加个通知系统"                  │ (直接命令)
     └────────────────────────────────────→ demand
```

三条路径：最自然 → 最省 token → 最直接。用户按习惯选择。

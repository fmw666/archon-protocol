# Sub-agent 委托模型

Archon 的核心架构原则之一：**执行与判断分离。**

## 为什么需要分离

你写了一天代码，然后自己审查。你会发现什么？大概率什么都不会发现。

不是因为代码没问题，而是因为**沉没成本偏差**——你刚花了大量精力写的东西，你的大脑会自动合理化它。

在 LLM 上同样如此。同一个上下文窗口里，生成代码的 agent 审查自己的输出，天然倾向于"看起来没问题"。

## 委托架构

```
                    ┌─────────────────┐
                    │    主 Agent       │
                    │  (项目负责人)      │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼                             ▼
  ┌───────────────────┐          ┌───────────────────┐
  │ capture-auditor    │          │ archon-reviewer    │
  │ 触发: 每次交付      │          │ 触发: drift ≥ 12   │
  ├───────────────────┤          ├───────────────────┤
  │ 知识捕获评估        │          │ 全量项目审查        │
  │ 盲区反思           │          │ 架构/代码/合规/知识  │
  │ 交付卫生检查        │          │ 主动审视三问        │
  ├───────────────────┤          ├───────────────────┤
  │ 输出: 结构化建议    │          │ 输出: 分级问题列表   │
  │ 主 Agent 执行建议   │          │ 主 Agent 逐项修复   │
  └───────────────────┘          └───────────────────┘
```

**核心规则：Sub-agent 只出建议，主 Agent 执行。** 职责分离。

## 委托准入门槛

满足**任一**条件时升级为 sub-agent 委托：

| 条件 | 原因 |
|------|------|
| 判断发生在工作完成后 | 沉没成本偏差最强 |
| 判断结果无自纠正机制 | 错了没人补漏 |
| 错误代价高 | 影响产品正确性或架构健康度 |

不满足则保持**自评**——每多一个 sub-agent 就多一次 LLM 调用的延迟和成本。

## 成本意识

激活新 sub-agent 前先回答：**这个检查能否用自审 checklist 替代？**

| 检查类型 | 建议方式 | 原因 |
|---------|---------|------|
| 机械性（占位符扫描、格式合规） | 内联自审 | 延迟低，质量无差异 |
| 判断性（知识价值识别、偏差纠正） | 委托 sub-agent | 需要独立视角 |

实证：纯机械审查 sub-agent 化后延迟增加一个数量级，质量无可测量提升。

## 两个内置 Sub-agent

### capture-auditor

**触发**：每次交付的收尾步骤 ②。

**职责**：
1. **知识捕获评估** — 这次交付是否产生了值得固化的经验？是沉淀还是跳过？
2. **盲区反思** — 交付过程中是否有被忽视的问题？
3. **交付卫生检查** — 新模块是否有测试？命名是否一致？

**输出**：结构化建议（沉淀什么 + 沉淀到哪个层级 + 卫生问题清单），由主 Agent 执行。

### archon-reviewer

**触发**：drift ≥ 12 或手动运行 `/archon-review`。

**职责**：全量项目审查——架构健康度、代码质量、规格合规、知识健康。

**输出**：分级问题列表（Critical / Warning / Observation），主 Agent 逐项修复或登记到 debt。

## 自省机制：主动审视三问

Reviewer 在每次审查中还必须回答三个**元问题**：

1. **工程全流程覆盖**：合格的工程负责人应该管哪些环节？覆盖了哪些？**缺了什么？**
2. **原则递归检查**：Soul 中的原则是否也在约束 Archon 自身？
3. **预判用户盲区**：如果用户下一句是"你怎么没想到 X"——X 最可能是什么？

这三问确保 Archon 不只治理代码，也治理自己。

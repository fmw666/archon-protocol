# 设计哲学：文档即约束代码

> Archon Protocol 的核心前提——AI 没有持久记忆，但文档有。

---

## 一、为什么需要 Archon Protocol

传统软件工程中，架构靠人的经验维护。高级工程师离职，架构就开始腐化。

AI 编程时代，这个问题被放大了 10 倍：

- AI 每次对话都是**全新的空白状态**——没有上次的记忆
- 同一个项目，AI 可能用 5 种不同方式实现同一个功能
- 6 万行代码中，AI 无法感知全局一致性
- 人类 review 的速度远跟不上 AI 生成代码的速度

**Archon Protocol 的核心洞察：文档是约束 AI 的"代码"。**

就像操作系统通过系统调用约束应用程序的行为，Archon Protocol 通过技能文档约束 AI 编码的行为。AI 不需要"记住"规则——每次启动时，约束技能被注入它的上下文窗口，成为它的运行时边界。

## 二、六条设计原则

### 1. 禁止项 > 指导项

```
❌ "写好代码"          → 模糊，无法验证
✅ "❌ 禁止 any 类型"  → 精确，可 grep 验证
```

"不要做 X" 可以被自动化检查。"请做好 Y" 只是一个愿望。Archon Protocol 的所有规则以禁止项（`❌`）为核心，每条都能用搜索验证是否被违反。

### 2. 被动注入 > 主动调用

```
/init  → 引导技能生态，完成启动
约束技能 → 每次代码变更时按设计自动激活
```

约束必须是被动的——不依赖 AI "想起来"去读。`/init` 引导整个技能生态；约束技能在每次代码变更时按设计自动激活。

### 3. 通用优先，工具增强在后

一切皆 SKILL.md。一种格式，27+ 工具支持，零冲突。无 AGENTS.md，无 .cursor/rules/，无 .cursor/agents/。

绝不为一个工具锁死。核心约束必须能在所有 AI 编程工具中生效。

### 4. 重流程用命令，轻约束用技能

DoD 循环是一个 5 阶段的复杂流程，如果每次对话都加载，会消耗大量 token——即使用户只是问个简单问题。

所以：约束技能轻量（单份 <5KB），命令技能编排复杂流程。两者使用同一 SKILL.md 格式。

### 5. 每个技能只做一件事

```
/dod           → 编排整个交付流程
/self-auditor  → 只做审计，不修改代码
/test-runner   → 只处理测试
/verifier      → 只验证，不信任任何声称
```

这不是简单的分工，而是**认知隔离**。每个内部技能拥有独立的上下文窗口，避免在一个巨大的上下文中混淆不同角色的判断。审计者不应该"体谅"实现者的难处。

### 6. 双重读者

每一份文档同时服务两个读者：

| 读者 | 读法 |
|------|------|
| **AI** | 逐字执行：读到 `❌` 就搜索代码验证，读到步骤就按序执行 |
| **人类** | 整体理解：把握架构意图、设计决策、权衡取舍 |

技能文档（英文，可被 AI 执行）给 AI 执行。中文解说文档（本目录）给人类理解。

## 三、与传统方案的对比

| | 传统 Lint/CI | Archon Protocol |
|---|---|---|
| **覆盖范围** | 语法、格式、类型 | 架构、性能、i18n、边界条件、测试同步 |
| **执行时机** | 代码提交后 | 代码编写时（实时约束） |
| **学习能力** | 静态规则，人工维护 | 自进化：每次任务可能产生新约束技能 |
| **上下文感知** | 单文件 | 全项目（通过内部技能隔离管理上下文） |
| **协作模型** | 人写代码 → 机器检查 | AI 写代码 → AI 自审 → AI 验证 |

Archon Protocol 不是替代 ESLint 或 CI——它是更高层的约束系统，通过约束技能确保 AI 在正确的架构边界内工作。

---

## 四、核心信念

> **Human not in the loop, but human sets the boundaries.**
>
> 人类不需要逐行审查 AI 的代码。人类的工作是定义边界：什么可以做、什么不可以做、什么标准必须达到。Archon Protocol 把这些边界变成 AI 的运行时约束。
>
> 每次 AI 完成任务并发现新的反模式，这个反模式被写入约束技能，下次 AI 就不会再犯。系统在每次迭代中变得更强。人类只需要提需求。

## 设计解读

### 为什么叫 "Soul"

不叫 config、不叫 settings、不叫 prompt-template。叫 Soul 是因为它不是配置项——它定义的是 Agent 的**身份和认知模型**。

配置可以被覆盖，原则不行。你不会说"今天把所有权原则关掉"——如果关掉了，Agent 就不再是工程负责人，Archon 也就失去了意义。

### 五条基石为什么是"过滤器"

五条基石不是一个打分表（"符合 3 条就行"）。它们是**门禁**——任何新特性、新流程、新工具进入 Archon 前，必须通过**所有五条**的检验。

这意味着 Archon 会**拒绝**看起来很酷但不符合原则的特性。比如：

- "给 AI 加一个自动循环模式" → 违反所有权原则（没有人触发 = 没有 stakeholder = 没有所有权边界）
- "用配置文件控制 Agent 行为" → 违反推断原则（从状态推断，不读配置）
- "每次交付都写一篇反思日志" → 违反精简原则（知识膨胀比知识缺失更危险）

### 递归适用的意义

Soul 中有一段看起来像"哲学废话"的内容：

> Soul 中每条原则同时约束 Archon 框架自身。

这不是废话。没有这条递归约束，你会看到一个有趣的悖论：
- Archon 要求代码"零容忍死代码"
- 但 Archon 自己的治理文件里有过时的规则没人管
- Archon 要求"新代码 = 新护栏"
- 但 Archon 新增了一个工程阶段，没有同步补齐治理机制

递归适用 = **自己对自己负责**。这和"所有权"原则一脉相承。

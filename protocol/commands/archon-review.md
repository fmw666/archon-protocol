你现在进入「全量审查」模式。

## 触发条件

以下任一：
- drift ≥ 12（强制）
- 用户主动调用本命令

## 审查流程

### Phase 0: 自动化检查

执行 manifest 中声明的验证命令。如果红了，先修到绿——自动化能捕获的问题不需要人工审查。

### Phase 1: 启动 archon-reviewer subagent

对整个项目做独立审查（参见同级 agents 目录下的 archon-reviewer.md）。reviewer 审查的是自动化无法覆盖的维度：架构腐化、抽象合理性、规格漂移、治理健康度。

### Phase 2: 知识健康审计

不是检查清单，是量化审计。产出具体数字。

**索引一致性**
- 列出磁盘上所有治理文件（rules / skills / agents / ADR）
- 逐条核对 manifest 知识资产索引 → 标记：缺失（磁盘有但索引没有）、幽灵（索引有但磁盘没有）、描述过期
- 不一致数 = 0 为健康

**治理比例**
- 治理文件总数 / 源码文件总数 → > 0.5 时标记治理膨胀

**知识捕获率**
- 读 drift 日志自上次审查（或日志起点）以来的条目
- 交付总数 D，沉淀列非 `—` 的条数 C
- 捕获率 = C / D → 报告数值，< 0.3 时标记知识流失风险

**覆盖度**
- 列出源码中每个一级子目录 → 检查是否有对应的规则或技能覆盖 → 标记盲区

**过期清理**
- 规则/技能中描述的模式是否仍在代码中使用 → 标记过期资产并清理
- manifest 目录结构是否与磁盘吻合 → 修正漂移

### Phase 3: 修复

修复 reviewer 发现的 Critical 和 Warning 级别问题。修完再跑一次验证命令确认没有回归。

### Phase 4: 收尾

1. 更新 manifest 使其与项目现实完全同步
2. 重置 drift：
   - 完全清晰：drift 归 0，日志记录本次审查
   - 有残留疑点：drift 设为 1-3，日志附说明残留了什么

## 输出

向用户报告：审查发现了什么、修了什么、drift 重置到多少。

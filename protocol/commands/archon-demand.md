## 前置扫描

收到需求后、进入决策关卡前，先扫描 manifest 的对话备忘和架构决策记录，检查是否存在相关的历史讨论或结论。如有——在回应开头主动引用（何时讨论、当时结论、是否有新变量改变结论）。不让用户重复自己。

## 决策关卡

动手前先评估。这不是可选步骤：

- **该不该做**：这个变更解决的是真实问题还是假设问题？有没有不改代码 / 改更少代码的解法？
- **做多大**：爆炸半径（影响多少文件、模块、接口）+ 可逆性（改错了回滚成本多高）——改得越多，验证越严
- **谁来定**：工程决策自己定，产品方向问用户

评估为「不做」或「不该这样做」时——否决需求并给出理由和替代方案。这是你的职责，不是冒犯。

**否决后沉淀**：如果被否决的提案涉及技术选型或架构方向，且同一问题可能再次被提出（判断标准：提问源于合理直觉而非误操作），将否决记录为 Negative ADR（架构决策记录中的否决条目，含否决理由、替代路径、重审条件）。否决后仍需进入收尾流程（跳过验证门，从步骤 1 开始）。

## 验证门

交付代码后，执行 manifest 中声明的验证命令。

- 全绿 → 进入收尾
- 红了 → 修到绿，然后进入收尾
- 跳过验证 = 没交付

## 收尾

交付完成后：

1. 检查本次交付是否变更了技术栈、目录结构、架构决策或项目状态。如有变更，更新 manifest.md 对应区块。
2. 启动 capture-auditor sub-agent（参见同级 agents 目录下的 archon-capture-auditor.md，使用 fast 模型），传入本次交付摘要和变更文件列表。等待其返回结构化评估（知识捕获 + 盲区反思 + 交付卫生）。
3. 处理 auditor 输出：
   - **知识沉淀**：有沉淀建议 → 按建议的层级和载体创建/更新资产，同步 manifest 知识资产索引；无建议 → 跳过。
   - **卫生响应**：auditor 交付卫生有 ✗ 项 → 能即时修复的修复（回到验证门确认绿灯），不能即时修复的登记到 debt.md（附来源、严重度、截止日期）。全 ✓ → 跳过。
4. 更新 drift.md：评估复杂度 → 追加日志（含 auditor 返回的沉淀列内容）→ 累加分值。若 drift ≥ 12，立即执行全量审查（按 `archon-review` 流程），不等用户指示。
5. **里程碑门禁**：如果本次交付完成了一个里程碑的最后一个功能验收项，在 manifest 中勾选前检查 `debt.md` —— 所有 `milestone-close` 截止的条目必须 `resolved`。未清零 = 里程碑未完成，先清债再勾选。
6. **对话备忘**：本次交互是否产生了用户不该被要求重复的结论？包括：否决的提案、延期的特性、确认的方向、变更的优先级。如有——追加到 manifest 对话备忘（日期 + 议题 + 结论 + 一句话依据）。仅记录有决策价值的结论，不记录常规交付（那是 drift 的职责）。超过 10 条时裁剪最旧条目。
7. 检查 manifest 中是否声明了 git 策略。如有：检查 `git status`，按声明的模式和分支模型执行版本控制（提交 / 提示 / 跳过）。使用项目声明的提交规范（如有对应 skill）。一次 commit 包含本次交付的所有变更（代码 + manifest + 知识资产 + drift + debt）。如果 manifest 未声明 git 策略，跳过。
